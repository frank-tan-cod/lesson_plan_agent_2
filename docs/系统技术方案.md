# 教学协助智能体系统技术方案

## 1. 文档说明

本文档基于当前仓库中已经实现并对用户开放的功能编写，遵循“有则详述、无则不编”原则。文档重点覆盖：

- 当前系统面向用户开放的功能模块
- 各模块的核心作用
- 前后端实现原理
- 关键技术细节、数据流与存储机制
- 当前实现边界与未闭环部分

本文不对仓库中未形成用户可用闭环的能力作扩展性编撰，也不将纯占位 UI 视为完整业务能力。

## 2. 系统总体概述

本系统是一个面向教学场景的智能工作台，核心目标是帮助教师围绕“教案”和“演示文稿（PPT）”进行创建、编辑、知识沉淀和导出。系统采用前后端分离架构：

- 前端基于 `Next.js 14 + React 18 + TypeScript + Tailwind CSS` 构建，负责认证页面、工作台页面、知识库页面、偏好设置页面以及文档编辑器交互。
- 后端基于 `FastAPI + SQLAlchemy + SQLite` 构建，负责业务接口、用户认证、文档持久化、AI 调用、知识库索引、流式编辑和导出服务。
- AI 能力通过 OpenAI 兼容接口接入 DeepSeek 系列模型，用于教案初稿生成、PPT 初稿生成、偏好解析、编辑器任务规划和知识问答整理。
- 知识库检索采用 `sentence-transformers + ChromaDB`，实现文档/图片的向量化索引与语义检索。
- 课件、图片、小游戏 HTML 等文件以本地文件方式保存到 `uploads/` 目录，并由 FastAPI 以静态资源方式对外提供访问。

从用户视角看，当前系统已形成以下完整功能闭环：

1. 注册、登录、退出与会话鉴权
2. 教案 / 演示文稿列表管理
3. 新建教案 / 新建演示文稿
4. 教案编辑器与 PPT 编辑器
5. 知识库管理、搜索与基于资料回答
6. 全局偏好预设管理
7. 个人中心基础功能

## 3. 总体技术架构

### 3.1 架构分层

系统整体可分为四层：

| 层级 | 组成 | 主要职责 |
| --- | --- | --- |
| 表现层 | Next.js 页面与组件 | 页面渲染、表单交互、编辑器预览、文件上传、SSE 流式消息消费 |
| 接口层 | FastAPI Router | 提供 REST API、SSE 编辑接口、导出接口、静态文件访问 |
| 业务层 | Service + Tool + Planner | 文档生成、编辑执行、知识检索、偏好管理、导出渲染、会话记忆 |
| 数据层 | SQLite、ChromaDB、uploads 目录 | 结构化数据持久化、向量索引、文件存储 |

### 3.2 前后端通信方式

系统当前使用两类通信方式：

- 常规业务操作使用 HTTP JSON API，例如登录、创建文档、查询知识库、保存偏好、更新演示文稿等。
- 编辑器对话使用 HTTP 长连接下的 SSE（Server-Sent Events）流式返回。前端通过 `fetch + ReadableStream` 解析事件流，后端通过 `StreamingResponse` 持续输出 `status`、`tool`、`tool_result`、`delta`、`follow_up`、`confirmation_required`、`done` 等事件。

这种设计使编辑器既能实时展示 AI 分步处理状态，又能在需要补充信息或确认破坏性操作时中断并恢复流程。

### 3.4 核心数据对象

系统围绕以下核心对象建模：

| 对象 | 作用 |
| --- | --- |
| `User` | 平台用户账号 |
| `Plan` | 统一存储教案和演示文稿，依靠 `doc_type` 区分 |
| `Conversation` | 编辑器会话，保存会话状态、最近轮次、临时偏好、待确认任务等 |
| `Operation` | 工具执行日志，用于编辑器可追溯和会话摘要 |
| `Savepoint` | 文档快照，用于回退恢复 |
| `KnowledgeFile` | 知识库文件元数据，支持文档与图片两类 |
| `PreferencePreset` | 全局偏好预设 |

其中，教案与 PPT 采用“同表存储”的实现方式：`Plan` 表承载两类文档，`doc_type=lesson` 表示教案，`doc_type=presentation` 表示演示文稿。这样可复用统一的列表、删除、更新时间、用户隔离等基础能力。

## 4. 用户认证与会话管理模块

### 4.1 核心作用

该模块负责用户注册、登录、退出、当前登录态识别，以及后续所有业务接口的身份隔离。

### 4.2 实现原理

后端认证流程如下：

1. 用户注册时，后端校验用户名唯一性，并使用 `passlib + bcrypt` 对密码进行哈希存储。
2. 用户登录时，后端校验用户名与密码，生成带 `sub=user_id` 的 JWT。
3. JWT 通过 `httpOnly Cookie` 写入浏览器，Cookie 名为 `lesson_plan_agent_session`。
4. 后续接口通过 FastAPI 依赖项统一解析 Bearer Token 或 Cookie 中的 JWT，得到当前用户 ID。
5. 各业务服务在查询和写入时均显式带入 `user_id` 过滤条件，实现多用户数据隔离。

前端认证流程如下：

1. `AuthProvider` 在应用启动时调用 `/api/user/me` 探测登录态。
2. 若已登录，则把非敏感用户信息缓存到 `localStorage`，用于提升首屏体验。
3. 受保护页面由 `AppShell` 检查认证状态，未登录时自动跳转到登录页。
4. 登录、注册成功后跳转到文档工作台。

### 4.3 技术细节

- JWT 算法：`HS256`
- 登录状态有效期：7 天
- Cookie 属性：`httpOnly`，`SameSite` 和 `Secure` 可通过环境变量控制
- 认证中间依赖：`get_current_user_id`
- 安全前置校验：服务启动时会校验 `JWT_SECRET_KEY` 是否为真实安全值，避免示例密钥上线

### 4.4 当前边界

个人中心中的“修改密码”目前只有前端两步式交互：

- 第一步通过重新调用登录接口校验旧密码
- 第二步仅完成前端校验和提示

当前仓库中未实现后端“修改密码”接口，因此这部分属于前端演示流程，不是完整闭环的账号安全修改能力。

## 5. 文档工作台模块

### 5.1 核心作用

文档工作台用于统一管理教案和演示文稿，支持列表查看、按文档类型切换、按标题关键词搜索、按学科/年级筛选、进入编辑器和删除文档。

### 5.2 实现原理

前端 `DocumentsView` 提供统一列表页，允许在 `lesson` 与 `presentation` 两种类型之间切换。搜索逻辑分为两类：

- 无关键词时调用 `/api/plans`，由后端按 `doc_type`、`subject`、`grade` 过滤
- 有关键词时调用 `/api/plans/search`，先按标题搜索，再在前端补充学科与年级的二次过滤

删除操作调用 `/api/plans/{plan_id}`，后端直接删除对应记录。

### 5.3 技术细节

- 列表数据来源：`PlanService.list / search`
- 查询排序：按创建时间倒序
- 文档类型切换：通过 `doc_type` 字段统一控制
- 文档列表页只展示用户自己的文档，底层查询统一带 `user_id`

## 6. 新建文档模块

### 6.1 核心作用

该模块支持两类文档新建：

- 新建教案
- 新建演示文稿

对于新建教案，系统支持结合教学要求、补充课程内容和知识库参考资料生成教案初稿；对于新建演示文稿，则创建一个空白 PPT 项目供后续编辑。

### 6.2 教案创建实现原理

当用户创建教案时，前端会提交：

- 标题
- 学科
- 年级
- 教学要求
- 补充课程内容
- 选中的知识库文件 ID 列表

后端 `PlanService.create` 处理逻辑如下：

1. 若用户没有填写教学要求，则创建一个空教案骨架：`content = {"sections": []}`
2. 若填写了教学要求，则先通过 `build_reference_context` 读取选中知识库文件内容
3. 再将教学要求、标题、学科、年级、补充课程内容、知识库参考上下文一并送入大模型
4. 要求模型只输出 JSON，结构中必须包含 `title`、`sections`、`metadata`
5. 后端对模型输出做规范化，保证章节格式、时长字段和元数据结构稳定
6. 最终写入 `plans` 表

### 6.3 演示文稿创建实现原理

若用户直接新建演示文稿，后端不调用生成模型，而是创建统一的 PPT 数据骨架：

- `title`
- `classroom_script`
- `slides`

后续由编辑器或“基于教案生成 PPT”功能进一步填充。

### 6.4 参考资料联动

新建教案页面同时集成了知识库联动能力：

- 可直接读取已有知识库文件并勾选参与本次创建
- 可在新建流程中即时上传文档或图片到知识库
- 上传成功后自动勾选到当前创建流程

这使知识库既是独立模块，又能在创建阶段作为上下文增强来源。

### 6.5 自动入库机制

文档创建完成后，后端会尝试将新生成的教案或演示文稿自动转为 Markdown 并写入知识库，用于后续检索和复用。这一机制由 `plan_ingestion.py` 提供。

## 7. 教案编辑器模块

### 7.1 核心作用

教案编辑器是系统最核心的交互模块之一，支持用户通过自然语言对话修改教案内容，同时在左侧实时预览当前教案结构，并提供回退点、保存进知识库、导出、生成小游戏、生成 PPT、临时偏好等围绕编辑过程的增强功能。

### 7.2 交互机制

编辑器不是简单的一问一答，而是一个“对话式任务执行系统”。整体流程为：

1. 前端进入编辑器时加载文档详情、历史会话、临时偏好和回退点
2. 若当前文档没有活跃会话，则自动新建会话
3. 用户输入修改需求后，前端调用 `/api/editor/chat`
4. 后端以 SSE 流的形式持续返回：
   - 当前状态
   - 规划出的工具调用
   - 工具执行结果
   - 流式自然语言回复
   - 追问请求
   - 确认请求
   - 最终完成结果

### 7.3 编辑器后端执行原理

教案编辑器核心由 `DocumentEditor` 负责调度，内部包含以下关键部件：

- `EditorPlanner`：负责把用户自然语言转换成结构化任务列表 `TaskList`
- `EditorGuardrails`：负责校验工具参数、阻止把“过程性占位文案”错误写入正文
- `ToolsRegistry + ToolExecutor`：负责注册并执行教案编辑工具
- `ConversationService`：保存会话状态、临时偏好、待确认任务、最近轮次
- `OperationService`：记录每次工具调用，形成可追溯操作历史
- `ConversationExecutionRegistry`：保证同一会话同一时刻只存在一个活跃执行流，并在新请求到来时取消旧请求

这套机制使编辑器具备以下能力：

- 可以先定位，再修改
- 可以在信息不足时主动追问
- 可以在删除、覆盖等高风险修改前先请求确认
- 可以把一次复杂需求拆成多步串行完成

### 7.4 教案编辑工具能力

当前已实现的教案编辑工具包括：

- `rewrite_section`：整体重写章节
- `adjust_duration`：调整章节时长
- `insert_element`：插入提问、案例、板书等教学元素
- `insert_section`：新增完整章节
- `delete_section`：删除章节
- `move_section`：移动章节顺序
- `add_image_placeholder`：插入图片占位符
- `search_in_plan`：全文搜索关键词
- `get_section_details`：读取章节细节
- `get_text_context_in_plan`：按原句定位文本上下文
- `replace_text_in_plan`：批量替换文本
- `replace_paragraphs_in_section`：按段落替换
- `insert_paragraphs_in_section`：按段落插入
- `delete_paragraphs_in_section`：按段落删除
- `evaluate_plan_suitability`：评估教案难度和入门适配度

这些工具都不是直接操作富文本编辑器 DOM，而是作用于后端存储的结构化教案 JSON，再统一落库并回推给前端预览。

### 7.5 会话追问与确认机制

若用户需求缺少关键条件，编辑器不会盲猜，而是通过 `ask_follow_up` 工具返回结构化追问：

- 问题文本
- 可选按钮
- 已完成步骤
- 待继续任务

若即将执行删除、覆盖等可能有破坏性的操作，则通过 `request_confirmation` 工具生成确认请求，前端展示“确认执行 / 取消本次修改”按钮，用户确认后才继续执行。

### 7.6 临时偏好注入

教案编辑器支持会话级临时偏好。其特点是：

- 只作用于当前会话
- 与全局偏好共同生效
- 通过结构化字段存入 `Conversation.metadata.temp_preferences`

支持的维度包括：

- 教学节奏
- 互动强度
- 内容展开程度
- 表达风格
- 视觉呈现倾向
- 其他要求

后端会把这些结构化偏好转换成中文提示语，作为编辑器规划和内容生成时的附加上下文。

### 7.7 回退点与知识快照

教案编辑器包含两条快照链路：

#### 回退点

- 保存当前文档内容到 `savepoints` 表
- 可在编辑器中查看、恢复和删除
- 仅用于编辑器内回退，不写入知识库

#### 保存进知识库

- 在创建回退点的同时，把当前教案内容转为 Markdown 文档
- 以 `editor_snapshot` 来源写入知识库
- 后续既可在知识库检索，也可通过回退点恢复

这种设计同时满足“内部版本回滚”和“知识沉淀复用”两类需求。

### 7.8 教案导出

教案编辑器支持导出为：

- Word（DOCX）
- PDF

导出机制由 `ExportService` 实现，导出时会：

1. 读取结构化教案内容
2. 将章节和文本块规范化
3. 使用 `python-docx` 渲染 DOCX
4. 使用 `reportlab` 渲染 PDF
5. 返回可下载的二进制流

对于教案中的小游戏，导出服务会自动把小游戏信息整理成“课堂小游戏”章节附加到导出文档中。

## 8. 小游戏生成模块

### 8.1 核心作用

小游戏生成模块面向教案场景，用于从已有教案中提取适合课堂互动的知识点，生成可直接预览和复用的 HTML5 小游戏。

### 8.2 实现原理

当用户点击“生成小游戏”时：

1. 前端调用 `/api/plans/{plan_id}/generate-games`
2. 后端从教案章节中提取内容
3. 优先调用大模型生成结构化游戏 JSON
4. 若模型不可用或返回失败，则退回到规则兜底生成
5. 生成结果统一规范成 `MiniGamePayload`
6. 为每个游戏渲染静态 HTML 文件，保存到 `uploads/games/`
7. 把游戏列表写回教案内容 `content.games`

### 8.3 支持的游戏模板

当前仅支持三类模板：

- `single_choice`：单项选择题
- `true_false`：判断题
- `flip_cards`：翻卡片

系统未实现其他互动模板，因此技术方案中不扩展到未出现的游戏类型。

### 8.4 用户侧表现

- 教案预览区可直接嵌入小游戏 HTML 页面进行预览
- 生成 PPT 时，小游戏会被自动识别并转化为对应入口页
- HTML 小游戏由静态文件直接提供访问，无需额外前端路由

## 9. 基于教案生成 PPT 模块

### 9.1 核心作用

该模块用于把教案内容自动转换为课堂展示型演示文稿，而不是简单把教案原文逐页复制到 PPT。

### 9.2 实现原理

PPT 生成流程由 `presentation_generator.py` 实现，核心步骤如下：

1. 读取源教案
2. 读取用户附加的知识库资料
3. 读取用户补充的课程内容
4. 读取生成风格配置，包括主题、密度、学校名称、Logo
5. 组装大模型提示词，要求模型先生成 `classroom_script`，再生成 `slides`
6. 要求输出严格 JSON，字段包含：
   - `title`
   - `classroom_script`
   - `slides`
7. 生成完成后执行一次低温 refine，对页型、分页、图片页正文长度等做二次修正
8. 归一化为 `PresentationContent`
9. 若教案中已有小游戏，则自动把小游戏入口页补充进 PPT
10. 保存为新的 `doc_type=presentation` 文档

### 9.3 已注册模板与版式规则

当前 PPT 只支持三种已注册模板：

- `title_body`
- `title_body_image`
- `title_subtitle`

系统没有更多模板，因此当前 PPT 预览、AI 生成、局部编辑和 PPTX 导出都围绕这三种模板工作。模板定义集中在 `presentation_layouts.py`，其中同时定义了：

- 逻辑模板名
- 对应 PPTX 渲染布局名
- 每种模板的正文框区域
- 图片框区域
- 分页容量估计

### 9.4 风格参数

当前风格系统支持两个维度和两个品牌字段：

- 主题风格：`学院蓝 / 讲堂绿 / 晨光橙`
- 内容密度：`舒展大字 / 均衡 / 紧凑`
- 学校名称
- 学校 Logo

风格参数同时作用于：

- 前端预览分页
- 字号缩放
- 页面头部与封面色彩
- PPTX 导出渲染

### 9.5 自动入库

PPT 初稿生成后，后端会把生成结果转为 Markdown 自动写入知识库。这样后续知识搜索可命中历史 PPT 内容。

## 10. 演示文稿编辑器模块

### 10.1 核心作用

演示文稿编辑器与教案编辑器共享同一套对话式编辑框架，但编辑对象从“教案章节”切换为“幻灯片集合”。用户可以在预览右侧通过自然语言指令，驱动 PPT 进行局部或整体改写。

### 10.2 核心能力

当前 PPT 编辑器支持：

- 查看整套 PPT 概要
- 查询指定页面详情
- 搜索标题、正文、备注、课堂内容稿中的关键词
- 新增页面
- 删除页面
- 复制页面
- 移动页面
- 修改页面标题、正文、副标题、模板、图片说明、图片 URL、备注
- 整体替换整套 PPT 内容
- 切换页面版式
- 绑定小游戏入口

### 10.3 实现原理

PPT 编辑器使用独立的 `presentation_tools` 工具集，核心特点是：

- 所有修改都作用于结构化 `PresentationDocument`
- 页面索引统一使用从 0 开始的后端索引
- 在执行模板切换时，系统会自动同步移除不适配的图片字段
- 对“过程性说明、待补充文案、示意稿文案”有专门拦截，防止错误写入幻灯片正文

例如，当用户输入的目标文本仍像“先搜索后填充”“准备合并第 8 页到第 9 页”这类过程描述时，后端会拒绝写入，并要求继续补充最终展示内容。

### 10.4 页面配图管理

PPT 编辑器支持对单页图片进行精细管理：

- 可从知识库已有图片中选择
- 可上传本地图片
- 上传前在前端浏览器端完成裁剪
- 保存后将图片 URL 写回对应幻灯片

裁剪逻辑完全在前端 Canvas 中完成，输出新的裁剪文件后再上传知识库，避免把原始超大图直接用于页面展示。

### 10.5 Logo 管理

Logo 管理与页面配图共用一套图片资产选择器，但 Logo 额外强调：

- 使用方形裁剪
- 写入的是演示文稿风格元数据，而不是某一页正文
- 保存后同时影响预览和导出

### 10.6 PPTX 导出

演示文稿导出由 `export_pptx.py` 实现，核心过程为：

1. 读取结构化 PPT 内容
2. 根据当前风格解析颜色、字号、Logo、本地图片路径
3. 对超长正文做分页
4. 使用 `python-pptx` 逐页渲染
5. 按模板绘制背景、标题区、正文区、图片区
6. 若页面中有图片，则尝试按比例适配到图片区
7. 若页面中有链接，则在正文中生成可点击超链接
8. 输出 `.pptx` 二进制文件

这意味着系统当前导出的是“程序绘制型 PPTX”，而不是浏览器截图型课件。

## 11. 知识库模块

### 11.1 核心作用

知识库模块用于沉淀教师上传的文档、图片，以及系统自动生成或保存的教案/PPT/快照内容，并支持：

- 文件上传
- 文件列表与筛选
- 文件信息编辑
- 文件删除
- 语义搜索
- 基于资料回答问题

### 11.2 文件类型与来源

当前知识库支持两类文件：

- 文档：`pdf`、`docx`、`md`、`markdown`
- 图片：`png`、`jpg`、`jpeg`、`webp`、`gif`、`bmp`

文件来源分为：

- 手动上传
- 系统自动沉淀
- 编辑器快照

前端列表页可以基于文件类型和来源进行筛选。

### 11.3 文档入库实现原理

文档入库流程如下：

1. 后端将原始文件写入 `uploads/documents/`
2. 根据扩展名选择解析器：
   - PDF：`pypdf`
   - DOCX：`python-docx`
   - Markdown：`markdown + HTMLParser`
3. 抽取纯文本后，按固定窗口大小和重叠长度切片
4. 使用 `EmbeddingService` 对文本切片生成向量
5. 把切片写入 ChromaDB
6. 同时把文件元数据写入 `knowledge_files` 表

图片入库流程与文档不同：

1. 图片原文件保存到 `uploads/images/`
2. 不做 OCR
3. 仅对用户填写的图片说明文本做向量化索引
4. 图片尺寸、大小、扩展名等信息写入元数据

### 11.4 检索机制

知识库搜索采用“混合检索 + 可选 LLM 结果整理”策略：

1. 先进行向量语义检索
2. 同时对文件名、描述、全文和元数据做关键词检索
3. 将两类结果聚合到文件级别
4. 综合生成相关度分数
5. 若开启“LLM 整理”，则把候选结果交给模型进行重排和摘要

最终搜索结果除基础片段外，还会包含：

- 命中原因
- 文件来源
- 触发方式
- 文档类型
- 检索策略

### 11.5 基于资料回答

“基于资料回答”功能建立在搜索结果之上：

1. 先执行知识搜索
2. 对前若干命中文档读取正文或图片说明
3. 把问题和资料片段一起发给大模型
4. 要求模型只基于命中资料生成回答，并给出引用文件 ID
5. 若模型不可用，则退回到规则式摘要回答

因此该功能不是开放域问答，而是有知识库证据约束的检索增强式问答。

### 11.6 自动知识沉淀

系统会在以下时机自动把内容沉淀进知识库：

- 新建教案
- 教案导出
- 生成小游戏后的教案更新
- 创建演示文稿
- 基于教案生成 PPT
- 导出演示文稿
- 保存编辑器快照到知识库

自动沉淀统一通过 `plan_ingestion.py` 把结构化内容渲染成 Markdown，再走文档入库流程。

## 12. 全局偏好预设模块

### 12.1 核心作用

偏好预设模块用于管理教师长期稳定的写作与生成偏好，使其在编辑器和生成链路中持续生效。

### 12.2 偏好模型

系统当前支持的结构化偏好维度包括：

- 教学节奏
- 互动强度
- 内容展开
- 表达风格
- 视觉呈现
- 其他要求

每一份偏好预设既保存结构化字段，也保存转换后的提示词文本 `prompt_injection`。

### 12.3 实现原理

偏好预设的创建与编辑有两种入口：

#### 结构化填写

用户通过表单选择各维度后，前端实时生成“系统注入预览”，后端保存为预设。

#### 自然语言解析

用户输入自然语言描述后，后端调用大模型，把偏好解析成多个候选建议，每个建议包含：

- 名称
- 描述
- 结构化偏好
- 对应提示词
- 标签

用户可将建议一键填入表单，再保存为正式预设。

### 12.4 生效方式

全局偏好预设保存在 `preference_presets` 表，并支持激活/停用。编辑器运行时会读取当前用户所有激活预设，把其中的 `prompt_injection` 拼接成统一偏好文本，作为规划和生成时的附加上下文。

## 13. 个人中心模块

### 13.1 核心作用

个人中心目前承担的是“账户展示与辅助信息入口”角色，而不是完整的用户资料管理中心。

### 13.2 当前已实现内容

当前个人中心包括：

- 当前登录用户名展示
- 退出登录
- 使用指南
- 联系邮箱
- 版本消息弹窗
- 修改密码前端流程演示

## 14. 系统特色综述

### 14.1 面向教师真实工作流的一体化设计

从当前已实现的功能来看，本系统的突出特点并不在于“单点生成一份教案”，而在于围绕教师日常备课流程，构建了一条从资料准备、内容生成、过程修改、资源衍生到成果导出的连续工作链路。教师可以在同一系统内完成以下动作：

- 结合教学要求、补充课程内容和知识库资料生成教案初稿
- 通过对话式编辑器持续修改教案或演示文稿
- 基于教案自动生成课堂展示型 PPT
- 从教案中进一步生成课堂小游戏
- 将阶段性成果导出为 Word、PDF、PPTX 等正式材料
- 将优质内容自动沉淀回知识库，供后续复用

这种“生成 + 编辑 + 衍生 + 沉淀 + 导出”的闭环，减少了教师在多个工具之间来回切换的成本，使系统更接近教学内容生产工作台，而不是一次性内容生成器。

### 14.2 以知识库为中心的资料增强与经验沉淀

系统的一大亮点，是把知识库从“附件仓库”提升为贯穿备课全过程的核心支撑能力。

一方面，知识库可在教案创建阶段直接参与生成。教师既可以从已有资料中勾选文档、图片作为参考，也可以在创建过程中即时上传新资料并立刻纳入本次生成上下文。这样生成出的教案不是脱离材料的通用文本，而是更贴近教师现有课程资源和授课意图的初稿。

另一方面，系统又支持把新生成的教案、PPT、导出成果以及编辑器快照自动回写到知识库。这样一来，教师在使用系统过程中形成的每一份成果，都有机会转化为下一次备课的可检索资产。对于教学实践而言，这种能力的价值在于：

- 把零散资料、历史教案和课件逐步沉淀为个人教学知识库
- 让“过去做过的内容”在下一轮备课中可以被再次检索和调用
- 让教师经验从单次使用结果，转变为可积累、可复用的长期资产

可以概括为一句话：系统不仅帮助教师生成内容，也帮助教师沉淀自己的教学资源体系。

### 14.3 对话式编辑不是简单问答，而是可执行的结构化改稿

系统在交互方式上的创新点，主要体现在编辑器。当前编辑器并非把用户输入直接交给模型自由生成，而是采用“规划 - 工具执行 - 状态反馈 - 结果落库”的任务式编辑框架。

在这一框架下，系统能够：

- 先理解教师意图，再拆解为章节重写、段落插入、时长调整、页面移动等具体动作
- 在信息不足时主动追问，而不是盲目猜测
- 在删除、覆盖等高风险修改前先请求教师确认
- 通过工具执行日志、会话历史和回退点，让修改过程可追溯、可恢复

这意味着系统的优势不只是“会写”，更在于“会稳妥地改”。对于教学场景来说，这一点尤为重要，因为教师往往不是需要一份从零虚构的文本，而是需要基于现有内容做多轮、细粒度、可控的精修。

### 14.4 教案、PPT、小游戏之间形成联动式成果生产

当前系统已经形成了较明显的多成果联动能力，这也是其教学应用价值较强的部分。

首先，教案不是终点。系统可以基于教案内容进一步生成 PPT，并在生成过程中考虑课堂展示逻辑、页面模板、内容密度、学校名称和 Logo 等风格信息，使演示文稿更适合作为课堂讲授材料，而不是原文照搬。

其次，教案还可以进一步衍生出课堂小游戏。系统会从现有教案中提取适合互动的知识点，生成可直接预览的 HTML5 小游戏，并在后续生成 PPT 时自动补充对应入口页。这样，教师在完成教案编写后，可以进一步快速得到课堂互动资源，提升教学活动设计的完整度。

这种“以教案为源头，向课件与互动资源扩展”的能力，有助于教师把备课成果自然转化为课堂实施材料，减少重复整理、重复制作和重复录入的工作量。

### 14.5 个性化偏好持续生效，适应不同教师风格

系统不仅支持一次性生成内容，还支持教师把个人偏好稳定地注入到后续创作过程中。

在长期层面，教师可以通过全局偏好预设管理自己的教学节奏、互动强度、内容展开程度、语言风格和视觉倾向，这些偏好会在后续编辑器和生成链路中持续生效。在短期层面，教师还可以在具体会话中设置临时偏好，以适应某一节课、某一次公开课或某一种课件风格的特殊需要。

这种“双层偏好机制”的意义在于：

- 让系统输出更贴近教师个人教学习惯
- 避免每次都重复描述相同的风格要求
- 在共性风格稳定的基础上，保留针对单次任务的灵活调整空间

对于教学应用而言，这有助于让 AI 输出从“能用”进一步走向“像这位教师自己会采用的表达方式”。

### 14.6 可回退、可导出、可复用，提升教学落地可靠性

很多教学工具停留在“生成页面看起来不错”，但在真实使用中，教师更关心成果能否稳定保存、恢复、导出和再次调用。系统在这方面也形成了较完整的支撑。

- 通过回退点机制，教师可以在关键修改前保留快照，必要时一键恢复
- 通过“保存进知识库”，教师可以同时获得可恢复版本和可检索资料副本
- 通过 Word、PDF、PPTX 导出，教师可以把系统内成果直接转化为教学管理和课堂展示需要的正式文件
- 通过统一的结构化存储，后续教案、PPT、图片和小游戏都可继续参与下一轮编辑或检索

因此，系统的价值不仅在于提升生成效率，也在于提高教学内容从编辑到正式使用的可靠性和可迁移性。

### 14.7 对教学工作的实际帮助

综合当前已实现能力，系统对教学工作的帮助主要体现在以下几个方面：

- 减轻重复性备课劳动，把资料整理、初稿搭建、格式搬运等工作前移给系统处理
- 提高教案与 PPT 的一致性，使课堂目标、教学内容和展示材料之间更容易保持连贯
- 增强课堂互动设计效率，让小游戏等资源不再完全依赖教师手工额外制作
- 支持教师经验长期积累，让优秀案例、历史课件和阶段快照逐步沉淀为个人教学资产
- 提升修改过程的可控性，降低直接让模型改稿所带来的不可预期风险

总体来看，本系统的创新点主要不是某一个单独功能特别“炫”，而是把教师备课中的多个关键环节真正串联起来，形成了一个兼顾生成效率、编辑可控、知识沉淀和课堂落地的教学协助平台。

## 15. 文件存储、数据存储与索引设计

### 15.1 结构化数据存储

结构化业务数据存储在 SQLite 数据库 `lesson_plans.db` 中，主要表包括：

- `users`
- `plans`
- `conversations`
- `operations`
- `savepoints`
- `knowledge_files`
- `preference_presets`

### 15.2 文件存储

本地文件统一落在 `uploads/` 目录下：

- `uploads/documents/`：知识库文档与自动沉淀 Markdown
- `uploads/images/`：知识库图片、PPT 配图、Logo 资源
- `uploads/games/`：小游戏 HTML 文件

FastAPI 将 `uploads/` 挂载为静态目录，因此前端预览、PPT 导出和小游戏 iframe 访问都可直接引用其公开路径。

### 15.3 向量索引

知识库检索与会话摘要检索都使用 ChromaDB 持久化向量：

- 文档/图片索引按用户分 Collection 保存
- 会话摘要索引单独使用摘要 Collection 保存
- 默认嵌入模型为 `BAAI/bge-small-zh-v1.5`

这种设计兼顾了检索效果与本地部署可控性。

## 16. 安全与隔离设计

### 16.1 用户隔离

系统所有核心业务表均包含 `user_id` 字段，且服务层查询默认带上当前用户过滤条件。因此：

- 用户 A 无法读取用户 B 的文档
- 用户 A 无法读取用户 B 的知识库文件
- 用户 A 无法读取用户 B 的会话和回退点

### 16.2 认证安全

- 密码采用哈希存储，不存明文
- JWT 不暴露给前端脚本读取，优先通过 `httpOnly Cookie` 保存
- 服务启动时校验 JWT 密钥，避免使用默认示例值

### 16.3 编辑安全

编辑器层面对高风险修改做了两类控制：

- 参数校验：工具参数必须匹配严格 schema
- 操作确认：删除、覆盖等动作先进入确认态

此外，对于 PPT 编辑，还额外防止把“流程说明”“示意稿文案”“待补充占位文本”写入正式页面。

## 17. 当前实现边界与结论

### 17.1 已形成完整闭环的能力

- 认证登录
- 文档工作台
- 新建教案 / 新建 PPT
- 教案对话编辑
- PPT 对话编辑
- 知识库上传、检索、问答
- 回退点与知识快照
- Word / PDF / PPTX 导出
- 全局偏好与会话临时偏好

### 17.2 当前仍属于局部实现或演示性质的内容

- 个人中心修改密码仅前端演示，未打通后端持久化
- 个人中心消息通知、使用指南、联系我们为静态内容
- 编辑器中“语音输入”按钮当前未见后端识别链路与前端录音处理闭环，不应视为完整语音功能

### 17.3 总体结论

当前系统已经不是单一的“教案生成器”，而是一个围绕教学内容生产的智能工作台。其核心特色在于：

- 使用统一数据模型承载教案与 PPT
- 以知识库作为资料沉淀和复用中心
- 以 SSE 对话编辑器作为内容修改核心入口
